# ADR-0006 — OpenAPI schema as single source of truth

**Status:** Locked 2026-05-14

## Context

`galley/prose/lib/` is Python. The web reader in `galley/apps/web/` is TypeScript. Galley's heavy-compute workers (`services/python-workers/`) are Python. The book repo's automation (`make` targets, scripts) calls Python. Future tool families (speech, comics, video) will mix Python, TypeScript, Swift (menubar), and possibly Go (vale-style binaries).

When a prose detector needs GPU compute (Phase 7+ metaphor / paradox / foreshadow / knowledge-state), that compute runs in a Python worker exposed via HTTP. The CLI (Python), the web reader (TypeScript), and any future tool needs a typed client for that endpoint.

Two paths:

- **Hand-write dual clients,** keep them in sync manually.
- **Single OpenAPI schema** as the contract; generate language-specific clients from it.

## Decision

OpenAPI 3.1 schema at `galley/apps/api/openapi.yaml` is the single source of truth for HTTP contracts across galley.

- TypeScript clients in `galley/packages/api-client/` are generated by `openapi-typescript` (MIT).
- Python clients in `galley/prose/lib/_common/remote.py` (and equivalents in future tool families) are generated by `openapi-python-client` (MIT) or equivalent.
- Schema additions are reviewed; client regeneration is automatic and gated by the generator's exit code.
- Hand-written wrappers on top of generated clients are permitted for ergonomics, but the wire contract is never edited away from the schema.

Initial schema (Phase 7.5) ships with zero endpoints — establishing the structure and the regen pipeline. Real endpoints land alongside the GPU-tier work in Phase 9+.

## Consequences

- New endpoint = edit schema + regenerate; both clients stay aligned by construction.
- Schema becomes the documentation. Endpoint behavior, request/response shapes, and error codes are documented in one place that produces code.
- Versioning is explicit: `info.version` bumps trigger client regeneration in downstream packages.
- Tool families inherit the pattern: `galley/speech/`, `galley/comics/`, `galley/video/` all consume the same OpenAPI schema for their respective HTTP needs. The schema is namespaced by family (`/prose/...`, `/speech/...`, etc.).

## Notes

The OpenAPI choice is deliberately neutral on RPC frameworks (gRPC, tRPC, JSON-RPC were considered). OpenAPI is the most broadly-tooled for HTTP/REST and lines up with galley's existing TS-on-the-front, Python-on-the-back-and-CLI shape.

For non-HTTP boundaries (CLI ↔ subprocess, file-format contracts), OpenAPI is not the right tool. Subprocess invocations are documented in module docstrings; file formats are documented in `docs/BOOK-CONFIG.md` (per-book yaml) and JSON Schema files (for `.prose-metrics.json`).
